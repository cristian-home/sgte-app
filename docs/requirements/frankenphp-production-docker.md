# FrankenPHP Production Docker Setup

## Description

Prepare the application for staging and production deployment on a VPS running Dokploy. The current Sail container uses `php artisan serve` (development server) which is unsuitable for production. This requirement introduces Laravel Octane with FrankenPHP as the application server, a multi-stage production Dockerfile, and a staging compose file for supporting services.

The production container MUST run four processes via Supervisor:
- **Octane (FrankenPHP)**: Application server on port 8000
- **Horizon**: Queue worker
- **Reverb**: WebSocket server
- **SSR (Inertia)**: Node.js SSR server (`php artisan inertia:start-ssr`)

The current Sail dev environment MUST remain unchanged.

## Notes

- The `dunglas/frankenphp` base image includes FrankenPHP + Caddy, but when using `php artisan octane:start --server=frankenphp`, Octane runs FrankenPHP as a standalone worker (no Caddy routing). TLS is handled by Dokploy's reverse proxy (Traefik).
- The `--workers=auto` flag lets Octane auto-detect the optimal worker count based on CPU cores.
- The `--max-requests=1000` flag prevents memory leaks by recycling workers after 1000 requests.
- Node.js is installed in the production image solely for Inertia SSR. Only the runtime is needed, not npm or build tools.
- If Reverb needs to scale independently in the future, it can be extracted into a separate container by creating a second service in compose pointing at the same image with a different supervisor config.
- The Sail dev environment (`docker/8.5/`, `compose.yaml`) remains completely untouched.
